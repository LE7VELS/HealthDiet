# 后端需求

> 整体范围见 [`../PROJECT_CONTEXT.md`](../PROJECT_CONTEXT.md)，接口见 [`../API_CONTRACT.md`](../API_CONTRACT.md)，数据结构见 [`../DATA_MODEL.md`](../DATA_MODEL.md)。

## 1. 目标

使用 Go + Gin 和 MongoDB 完成真实可运行的饮食记录系统：

```text
注册登录
  → 保存档案和目标
  → 查询或创建 Food（食品或菜谱）
  → 保存饮食记录
  → 计算每日汇总
  → 返回最近 7 天趋势
```

后端请求先经过：

```text
Router → Middleware → Handler
```

Handler 后续按业务复杂度选择调用方式：

```text
简单 CRUD：Handler → Store → MongoDB
复杂业务：Handler → Service → Store → MongoDB
```

Handler 和 Service 都不得直接操作 MongoDB Driver。Service 只承载真实业务规则和流程，不为简单转发而创建。

## 2. 当前功能

### 认证

- 用户名、邮箱和密码注册。
- 用户名或邮箱登录。
- 密码只保存安全哈希。
- JWT 保护私有接口。

### 用户数据

- 读取和更新个人档案。
- 读取和更新营养目标。
- 读取和更新饮食偏好。
- 用户只能访问自己的数据。

### Food

- Food 使用统一模型，通过 `kind=food|recipe` 区分食品和菜品/菜谱；当前系统把菜品和菜谱视为同一类型，不建立独立菜谱业务。
- 查询 Seed、JSON/CSV 导入和当前用户自定义 Food；系统数据和用户数据都可以是食品或菜谱。
- 查看 Food 详情。
- 创建、修改和删除自己的自定义食品或菜谱。
- 菜谱原料引用当前用户可访问的基础食品并填写克数；当前不允许菜谱嵌套菜谱。
- 创建或修改菜谱时，由 Service 读取原料营养、保存计算快照，并计算总营养、每份营养和每 100g 营养。
- 提供成品重量时按成品重量计算；未提供时按原料总重量估算，并保存计算依据。
- 制作步骤保存为文本，但当前不根据步骤文本猜测营养损耗或油脂吸收。
- 任一原料缺少热量或宏量营养时拒绝生成菜谱营养并返回数据不完整错误；纤维缺失继续保留为未知值。
- Food 不足时允许创建自定义 Food，不能阻塞饮食记录。

### 饮食记录

- 创建、查询、编辑和删除记录。
- 每条记录包含时间、餐次、Food 条目和可选图片。
- 后端根据 Food 和重量或份量计算营养值。
- 保存营养快照，Food 后续修改不改变历史记录。

### 图片

- JPEG、PNG、WebP，单张最大 10 MB。
- 每条饮食记录最多一张图片。
- 图片由 Go 服务保存，MongoDB 只保存元数据。
- 图片不用于识别食物或估算重量。

### 营养结果

- 按日期返回每日营养汇总。
- 返回连续最近 7 天趋势。
- 返回热量和宏量营养的简单规则提示。
- 缺失数据必须明确说明，不能猜测为零。

## 3. 数据来源

Food 只来自：

1. Seed。
2. JSON 或 CSV 开发导入。
3. 用户创建或提交的自定义 Food，包括食品和菜谱。

当前不接入爬虫、公共食品 API 或 AI 数据。

## 4. 错误和安全

- API 使用统一 JSON 错误结构。
- 区分输入错误、未登录、资源不存在、数据冲突和内部错误。
- 后端日志记录请求失败的稳定错误码、人类可读安全消息、路径和必要请求元数据；未知内部错误记录完整包装错误链，便于定位 Store 或 Service 的具体失败位置。
- 日志不得记录明文密码、完整 JWT、数据库连接串或请求体中的敏感信息；登录失败不在日志中区分账号不存在和密码错误。
- 不向客户端返回密码哈希、数据库错误、连接串或服务器路径。
- 所有私有查询都带当前用户 ID。
- JWT 密钥、MongoDB 地址和存储配置来自环境变量。

## 5. 本地运行

- Gin 默认监听 `:8080`，可通过 `HTTP_ADDR` 修改。
- MongoDB 默认连接 `mongodb://127.0.0.1:27017` 和 `healthdiet` 数据库，可通过 `MONGODB_URI` 和 `MONGODB_DATABASE` 修改。
- API 启动时先连接并 Ping MongoDB，再幂等创建当前数据模型需要的集合和索引；连接或初始化失败时不启动 HTTP 服务。
- MongoDB 和 API 后续通过 Docker Compose 提供本地环境。
- 当前不建设健康检查、监控、追踪或复杂运维能力。

当前只保留 Router、Middleware、Handler、按需 Service、Model 和 Store 的简单结构；具体业务模块在实现对应功能时再加入，不提前创建无用空模块或接口。Router 按已实现模块拆分 `*_routes.go`，并通过带命名字段的依赖结构体完成装配；Model 只表示共享业务数据，不是请求必须经过的一层。

## 6. 手工验收

至少手工验证：

1. 注册、登录和无效 Token。
2. 档案、目标和偏好的保存读取。
3. 食品、菜谱查询和自定义 Food，并核对菜谱营养计算及估算标记。
4. 饮食记录新增、查询、修改和删除。
5. 图片格式、大小和归属校验。
6. 每日汇总和最近 7 天趋势。
7. 两个用户之间不能互相读取数据。

代码仍需通过 `gofmt`、`go vet` 和 `go build`。

## 7. 当前不做

- AI、LLM、Python Agent、Embedding 和向量数据库。
- 自动菜谱生成和完整推荐系统。
- 爬虫、图片识别和自然语言录入。
- 长期历史分析。
- 健康检查、自动化测试体系和复杂基础设施。

## 8. 后续 AI 和知识库扩展边界

当前不实现以下能力，但后端需要保持未来可扩展：

- 公共食品知识库与用户核心业务库隔离；未来 AI 可以使用独立只读凭证查询公共知识，不直接读取用户私有集合。
- 用户档案、目标、偏好、饮食记录、汇总和趋势继续由 Go API 按当前用户权限提供。
- 后续可以增加聚合的 Agent 上下文工具，一次返回完成问答或推荐所需的用户数据，避免大量细粒度接口调用。
- AI 只负责知识检索、自然语言解释和提出候选建议；营养计算、过敏原等约束过滤以及最终写入由 Go Service 完成。
- AI 发起的写操作必须经过用户确认，并复用现有 Service 业务规则。
- 爬虫数据进入清洗和校验流程后才能成为正式 Food，不能直接作为可信营养数据使用。
