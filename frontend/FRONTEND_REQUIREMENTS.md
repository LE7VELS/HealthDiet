# 智能膳食助手前端需求

## 1. 项目目标

建设一个同时支持电脑和手机的响应式 Web 应用。用户可以维护个人健康档案、查询食品与菜谱、记录饮食、查看营养统计和膳食建议。

当前阶段只实现前端：

- 不接入真实后端。
- 不引入 AI 识别或 AI 对话。
- 使用 Mock 数据演示完整交互。
- 不把前端 Mock 结果宣传为真实健康分析。

## 2. 用户主流程

```text
登录/注册
  → 完善个人档案
  → 搜索菜品或上传用餐图片
  → 手动确认菜名、重量、餐次和时间
  → 保存饮食记录
  → 查看营养报告
  → 查看膳食建议
```

## 3. 路由规划

| 路由 | 页面 | 访问要求 |
| --- | --- | --- |
| `/login` | 登录 | 公开 |
| `/register` | 注册 | 公开 |
| `/app/dashboard` | 首页概览 | 需登录 |
| `/app/meals` | 饮食记录 | 需登录 |
| `/app/meals/new` | 新建饮食记录 | 需登录 |
| `/app/foods` | 食品与菜谱库 | 需登录 |
| `/app/foods/:id` | 食品/菜谱详情 | 需登录 |
| `/app/reports` | 营养报告 | 需登录 |
| `/app/recommendations` | 膳食建议 | 需登录 |
| `/app/profile` | 个人档案 | 需登录 |

未登录访问受保护页面时跳转到 `/login`；登录成功后默认进入 `/app/dashboard`。

## 4. 全局布局

### 电脑端

- 左侧为可收起导航栏，顶部显示页面标题和用户菜单。
- 主内容区使用限制最大宽度的居中布局，避免大屏内容过度拉伸。
- 饮食记录和食品库优先使用表格或卡片列表。

### 手机端

- 改用顶部栏和底部导航，不展示常驻侧边栏。
- 底部导航保留：首页、记录、食品、报告、我的。
- 表格在小屏下转换为卡片列表，关键操作按钮尽量固定在容易点击的位置。
- 主要表单使用单列，避免水平滚动。

建议以 `768px` 作为手机/大屏布局的主要切换点，但组件应根据实际空间自适应，不应只依赖固定设备宽度。

## 5. 页面需求

### 5.1 登录

- 字段：用户名或邮箱、密码。
- 提供密码显示/隐藏、保持登录和前往注册的入口。
- 表单校验需明确提示必填、邮箱格式和密码长度问题。
- Mock 登录成功后保存演示会话，并跳转首页。

### 5.2 注册

- 字段：用户名、邮箱、密码、确认密码。
- 健康档案不强制在注册页一次填完，注册成功后再引导完善。
- 密码必须二次一致，并在客户端做基础强度校验。

### 5.3 首页概览

- 显示当天日期、问候语和档案完善提示。
- 显示今日热量、蛋白质、脂肪和碳水化合物概览。
- 提供明显的“添加饮食记录”主操作。
- 显示今日已记录餐次和最近记录。
- 显示一条简短膳食提示，并可跳转至建议页。
- 无任何记录时展示引导型空状态，而不是空白图表。

### 5.4 新建饮食记录

页面包含：

- 用餐日期与时间。
- 餐次：早餐、午餐、晚餐、加餐。
- 可选用餐图片。
- 菜品搜索与添加。
- 每个菜品的重量、份量和删除操作。
- 估算营养合计。
- 保存和取消。

当前不接入 AI，因此上传图片只作为用餐记录的附件，不自动生成菜名和营养数据。用户仍需手动添加菜品和重量。

#### 图片上传交互

- 电脑端支持点击选择和拖拽上传。
- 手机端支持拍照和从相册选择。
- 第一版每顿饭最多一张图片。
- 接受 JPEG、PNG 和 WebP，单张最大 10 MB。
- 选择后显示预览、文件名和大小，允许更换或删除。
- 不支持的格式或超限文件必须在上传前提示。
- Mock 阶段只保留当前会话内的预览 URL，不声称已永久保存。
- 接入真实后端后，前端通过统一 API 适配层提交图片，由服务端决定保存到本地文件系统或 OSS 等对象存储；前端不得内置 OSS 访问密钥或绕过服务端持久化边界。

### 5.5 饮食记录

- 按日期范围、餐次和关键词筛选。
- 默认按时间倒序，同一天内按早餐、午餐、晚餐、加餐分组。
- 每条记录显示时间、菜品、重量、图片缩略图和营养摘要。
- 支持查看、编辑和删除；删除必须二次确认。
- 编辑时允许修改时间、餐次、菜品和重量。
- 筛选无结果时展示空状态和清除筛选入口。

### 5.6 食品与菜谱库

- 支持按名称搜索和分类筛选。
- 分类至少包含：主食、肉蛋奶、蔬菜、水果、豆类、菜肴和饮品。
- 列表显示名称、分类、每 100g 热量和主要营养摘要。
- 支持分页或“加载更多”，手机端优先使用卡片列表。
- 详情页显示：图片、别名、标准份量、每 100g 营养数据、常用份量换算、原料和做法。
- 从详情页可直接跳转到新建记录，并预选当前菜品。

### 5.7 营养报告

- 可选择最近 7 天、30 天或自定义日期范围。
- 显示总热量、日均热量及蛋白质、脂肪、碳水化合物、膳食纤维的日均值。
- 折线图展示每日变化，对比图展示用户摄入与参考值。
- 分类展示主要营养素、矿物质和维生素，避免首屏同时堆叠过多图表。
- 数据不足时显示“暂无足够记录生成报告”，并提供添加记录入口。
- 所有结果标注为膳食参考，不使用“诊断”、“治疗”等医疗表述。

### 5.8 膳食建议

- 展示营养素状态：偏少、参考范围、偏多。
- 每项显示日均摄入、参考值和简短说明。
- 根据 Mock 规则展示建议菜品，包含菜名、图片、关键营养、建议份量和推荐原因。
- 支持喜欢、不喜欢和查看详情，反馈暂存于 Mock 数据层。
- 建议页不使用 AI 话术，不出现“AI 已为你分析”等误导性文案。

### 5.9 个人档案

- 基本信息：用户名、邮箱、出生日期或年龄、性别。
- 身体信息：身高、体重。
- 可选信息：特殊人群状态、健康限制、忌口和备注。
- 用户可以查看档案完善度并编辑。
- 修改前端 Mock 信息后给出成功 Toast，返回页面时保留修改结果。

## 6. Astryx 组件使用

- 应用框架优先使用 App Shell、Side Nav、Mobile Nav 和 Layout。
- 表单优先使用 Field、Text Input、Number Input、Selector、Date Input 和 File Input。
- 内容展示优先使用 Card、Table、Tabs、Badge、Empty State 和 Skeleton。
- 反馈使用 Toast、Banner、Dialog 和 Alert Dialog。
- 业务代码不应大量直接依赖 Astryx 导入路径，应尽量通过 `components/ui` 中的项目级组件使用。

## 8. Mock 与 API 边界

- 页面只调用 `lib/api` 暴露的函数，不直接导入 Mock JSON。
- 开发环境可通过 `VITE_USE_MOCKS=true` 使用 Mock 实现。
- Mock 函数应返回 Promise，并可模拟加载时间、空数据和错误。
- 未来接入 Go API 时保持函数签名和页面调用方式不变，只替换 API 适配层。

主要类型至少包含：

```ts
type UserProfile = { /* 账号和健康档案 */ }
type Food = { /* 食品、份量和营养数据 */ }
type MealRecord = { /* 餐次、图片、菜品和营养快照 */ }
type NutritionSummary = { /* 营养合计 */ }
type NutritionReport = { /* 日期范围、趋势和对比 */ }
type Recommendation = { /* 菜品建议和原因 */ }
```

## 9. 通用交互状态

所有数据页需要考虑：

- 首次加载：使用 Skeleton。
- 提交中：禁用重复提交，按钮显示进度。
- 空数据：说明原因并提供下一步操作。
- 错误：显示可理解的文案和重试入口。
- 操作成功：使用 Toast，不使用浏览器 `alert`。
- 危险操作：使用 Alert Dialog 二次确认。

## 10. 无障碍与响应式验收

- 表单控件具有可见标签和错误说明。
- 主要操作可使用键盘完成，对话框打开后焦点正确。
- 不只通过颜色表示营养状态，同时显示文字或图标。
- 手机端主要点击区域尺寸足够，不出现难以点击的密集小按钮。
- 至少验收 375px 手机、768px 平板和 1440px 桌面布局。

## 11. 开发完成标准

- 所有规划路由可访问，导航和返回行为正确。
- 核心流程可在 Mock 模式下完整演示。
- 图片选择、预览、替换、删除和校验可用。
- 电脑和手机布局无主要溢出、遮挡和无法操作问题。
- TypeScript 检查、lint 和生产构建通过。
- 代码中不包含密钥、真实用户数据或硬编码生产地址。
- 遵守 [FRONTEND_CONSTRAINTS.md](./FRONTEND_CONSTRAINTS.md) 中的技术和 Git 约束。
