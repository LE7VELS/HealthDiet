# 项目背景与开发上下文

> 本文档说明项目当前阶段、范围和技术边界。开始任务前先阅读本文档。

## 1. 项目背景

这是一个面向个人用户的饮食记录与营养分析系统，帮助用户记录饮食、查看每日营养摄入和最近趋势。

本项目只提供饮食参考，不提供医疗诊断或治疗建议。

## 2. 产品主链路

```text
用户注册登录
  → 建立个人档案和营养目标
  → 查询或创建 Food（食品或菜谱）
  → 添加饮食记录
  → 保存到 MongoDB
  → 查看每日营养汇总
  → 查看最近 7 天趋势
```

## 3. 当前开发阶段

当前同时开发 React 前端和 Go 后端，目标是完成一个本地可真实运行的小型系统。

- 使用真实 Go API，不以 Mock 演示作为完成标准。
- 数据保存到 MongoDB。
- 当前以手工功能测试为主。
- 开发时保持结构清晰，但不提前建设复杂基础设施。

## 4. 当前前端范围

- 登录和注册。
- Dashboard。
- 饮食记录。
- Food 搜索、详情和用户自定义 Food；Food 同时覆盖食品和菜谱。
- 每日营养汇总和最近 7 天趋势。
- 营养目标、饮食偏好和个人档案。
- 图片附件选择和上传。
- 响应式电脑端和手机端。

## 5. 当前后端范围

- Go + Gin API。
- JWT 注册登录和认证。
- 用户档案、营养目标和饮食偏好。
- Food 查询和自定义 Food；系统和用户都可以提供食品或菜谱类型的 Food。
- 饮食记录 CRUD。
- 图片上传和元数据。
- 每日营养汇总和最近 7 天趋势。
- MongoDB 持久化。
- Docker Compose 本地环境。

## 6. 当前食物数据方案

Food 数据只来自：

1. 项目 Seed 数据。
2. JSON 或 CSV 开发导入。
3. 用户创建的自定义 Food。

找不到所需食品或菜谱时，用户可以创建相应的自定义 Food，不能因为数据不全阻塞主流程。

当前统一使用 `Food` 表示可加入饮食记录的条目：

- `food`：单一食品或已有营养数据的普通食物。
- `recipe`：菜品和菜谱在当前系统中含义相同，表示包含原料、用量和制作步骤的组合 Food，也可参与搜索、选择和饮食记录。
- 两种类型都可以来自系统 Seed、JSON/CSV 开发导入或用户创建。
- 菜谱原料引用已有的基础食品并填写重量，Go Service 根据原料营养确定性计算菜品总营养、每份营养和每 100g 营养。
- 用户可以填写成品重量；未填写时暂用原料总重量估算每 100g 营养，并明确标记为估算值。
- 制作步骤用于记录和展示，当前不根据自然语言做法猜测烹饪损耗或油脂吸收。

## 7. 当前明确不做

- AI Agent、LLM、Python Agent、Prompt、Embedding 和向量数据库。
- 自动生成菜谱和完整个性化推荐。
- 食品网站爬虫和公共食品 API。
- 图片识别食物和自动估算重量。
- 自然语言录入。
- 原生移动 App。
- 医疗诊断和治疗建议。
- 30 天、自定义区间等长期分析。

## 8. 技术栈

### 前端

- React + TypeScript + Vite。
- Astryx Design System + StyleX。
- React Router。
- TanStack Query。
- React Hook Form + Zod。
- ECharts。

### 后端

MongoDB 不使用 GORM。图片由 Go 服务处理，MongoDB 只保存图片元数据和访问标识。

## 9. 当前整体架构

```text
React Web
   │ HTTP / JWT / 图片
   ▼
Router + Middleware
          ↓
       Handler
       ↙     ↘
 简单 CRUD   复杂业务
     ↓          ↓
   Store     Service
     ↓          ↓
     └──── Store
             ↓
          MongoDB
```

Handler、Service 和 Store 共享 `internal/model` 中的业务 Model；Model 是数据结构，不是请求必须经过的新层。

- React 只调用 Go API。
- Router 只注册路径和中间件。
- Middleware 处理日志、异常恢复和后续 JWT 认证。
- Handler 处理 HTTP 输入输出；没有业务编排的单集合 CRUD 可以直接调用 Store。
- Service 只用于认证、所有权和状态规则、跨集合流程、营养计算、汇总等复杂业务，不创建只做一行转发的 Service。
- Model 表示 User、Food、Meal、Nutrients 等业务数据，不依赖 Gin、MongoDB Driver 或 BSON。
- Store 负责 MongoDB 数据访问，Handler 和 Service 都不能直接使用 MongoDB Driver、BSON 或 Collection。
- 是否使用 Service 由业务复杂度决定，不要求每个请求机械地经过所有目录。
- 营养值来自 Food、用户确认的重量或份量和确定性公式。

## 10. 食物数据边界

- 当前不追求完整互联网食品库。
- Food 是统一模型，食品和菜谱不拆成独立集合或独立业务系统。
- 菜谱原料只能引用当前用户可访问的基础食品，当前不允许菜谱嵌套菜谱，避免循环依赖。
- 菜谱营养由后端根据原料快照、用量、份数和成品重量计算；前端提交的营养合计不可信。
- 制作步骤本身不直接参与当前公式，后续若增加烹饪损耗规则，仍必须使用可解释的确定性参数。
- 不猜测缺失营养值，缺失和真实零值必须区分。
- Food 修改不能悄悄改变历史饮食记录，记录保存时保留营养快照。
- 爬虫和公共食品 API 留到后续阶段。

## 11. 后续阶段

1. 扩展 Food 数据来源，增加爬取、公共数据导入、清洗、去重、来源追踪和质量管理。
2. 建立与核心业务数据隔离的公共食品知识库，保存食品资料、来源文档和可检索知识；是否使用 Embedding 或向量数据库在该阶段再决定。
3. 增加更长时间的历史分析和更多确定性规则提示。
4. 数据与规则稳定后，再增加食品知识问答、个人饮食问答、候选推荐和 AI Agent。

## 12. AI 协作规则

未来 AI 采用混合访问方式：

- 用户档案、目标、偏好、饮食记录和图片等核心业务数据必须通过 Go API 访问；复杂业务复用对应 Service。
- 为减少零散调用，Go API 可以提供一次返回档案、目标、偏好、近期记录、每日汇总和趋势的 Agent 上下文工具。
- AI 可以使用独立只读凭证访问与核心业务库隔离的公共食品知识库或只读视图，但不能访问用户私有集合。
- 爬虫和导入程序先保存、清洗、去重和校验原始数据，不直接把未确认数据写入正式 Food。
- 营养值、每日汇总、候选过滤和写入校验仍由 Go Service 使用确定性规则完成。
- AI 可以检索知识、调用受控工具、解释结果和提出操作草稿；涉及新增或修改用户数据时，必须经过用户确认并由 Go API 执行。
- AI 不得生成虚假 Food、绕过 Go API 写核心业务数据或提供医疗诊断和治疗建议。

以上是后续扩展边界，不属于当前阶段的实现要求；当前不预建 AI 服务、知识库集合、Embedding、向量数据库或 Agent 接口。
