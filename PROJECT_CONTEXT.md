# 项目背景与开发上下文

> 本文档用于帮助开发者和 AI 快速理解项目。开始工作前，请先阅读本文档、[FRONTEND_REQUIREMENTS.md](./FRONTEND_REQUIREMENTS.md) 和 [FRONTEND_CONSTRAINTS.md](./FRONTEND_CONSTRAINTS.md)。

## 1. 项目背景

本项目是一个面向个人用户的轻量级膳食管理助手。它希望解决以下问题：

- 用户不清楚自己每天吃了什么、吃了多少。
- 难以将日常饮食转换为直观的热量和营养摄入。
- 不知道当前饮食是否均衡，也不知道下一步应该如何调整。
- 食品成分、菜谱、饮食记录和营养报告缺少统一入口。

项目定位为日常饮食记录和营养参考工具，不是医疗诊断或治疗系统，不代替医生和专业营养师。

## 2. 产品主链路

```text
建立个人档案
  → 记录每日饮食
  → 计算营养摄入
  → 生成趋势和对比报告
  → 识别偏少或偏多的营养素
  → 给出可执行的膳食建议
```

长期核心功能为：

1. 账号与个人健康档案。
2. 食品与菜谱库。
3. 饮食记录，包含电脑和手机图片上传。
4. 营养计算与报告。
5. 膳食建议与反馈。
6. AI 膳食 Agent，作为后续阶段能力。

## 3. 当前开发阶段

当前只实现前端，目标是用 Mock 数据完成可演示、可交互的完整页面流程。

当前范围：

- 实现响应式电脑端和手机端界面。
- 实现登录注册、首页、饮食记录、食品库、营养报告、膳食建议和个人档案。
- 支持电脑选择/拖拽图片，以及手机拍照/相册选择。
- 图片当前只作为饮食记录附件，不做自动菜品识别。
- 所有页面通过统一 API 适配层使用 Mock 数据，为后续接入 Go API 保留清晰边界。

当前不做：

- 不开发真实后端。
- 不接入 MongoDB。
- 不接入 AI、图片识别或自然语言录入。
- 不开发原生手机 App，使用同一套响应式 Web 前端覆盖电脑和手机。

## 4. 已确定技术栈

### 前端

- React + TypeScript + Vite。
- Astryx Design System + StyleX。
- React Router。
- TanStack Query。
- React Hook Form + Zod。
- ECharts。

Astryx 处于 Beta 阶段，应固定依赖版本，并通过 `components/ui` 封装常用组件，避免业务代码与其 API 过度耦合。

### 后端（后续阶段）

- Go + Gin。
- MongoDB 官方 Go Driver v2。
- 图片由服务端统一处理；开发环境可保存到本地文件系统，部署后可切换到 OSS 等对象存储，MongoDB 只保存图片元数据和访问标识。

本项目计划使用 MongoDB，因此不使用 GORM。GORM 面向关系型数据库；MongoDB 应通过官方 Driver 访问。

### AI（更后期）

未来如果引入 AI，可增加 Python + FastAPI Agent 服务。AI 只负责理解用户输入、调用受控工具和解释结果，不直接操作数据库，不凭空生成营养数值。

## 5. 未来整体架构

```text
React Web
   │
   │ HTTP
   ▼
Go + Gin API
   ├── MongoDB
   ├── 图片存储
   └── Python Agent（后续可选）
```

边界原则：

- React 只调用 Go API。
- Go 负责账号、数据校验、饮食记录、营养计算和数据库访问。
- 营养数值来自食品数据库和确定性公式，不由 AI 猜测。
- 用户提交的菜名、重量和图片在保存前应允许确认和修改。

## 6. 新项目起点

- 本文档只总结当前已确认的新项目目标，不以任何旧代码作为实现参考或兼容目标。
- 新前端作为独立项目从根目录 `web/` 开始建设。
- 页面、数据类型、交互和未来 API 边界均以本次确认的文档为准，无需保持历史实现的结构和接口。

## 7. AI 协作规则

- 开发前先检查本文档和需求、约束文档。
- 未经用户要求，不扩大当前阶段范围，不提前实现后端或 AI。
- 修改前检查 Git 状态，保留用户现有改动。
- 项目根目录 `.gitignore` 必须保留和更新，不提交依赖、构建产物、环境变量、密钥和本地上传文件。OSS 接入代码和配置示例应正常入库，访问密钥、真实服务地址等敏感配置必须通过环境变量提供。
- 未经明确授权，不执行 commit、push、rebase、reset 或强制 Git 操作。
- 不覆盖或删除与当前任务无关的文件和改动。
